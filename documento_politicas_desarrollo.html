👔 Políticas de Oficina y Vestimenta

Lineamientos de Vestimenta

Se establecieron las normas de vestimenta para proyectar una imagen profesional:

  • Ropa en buen estado: Debe estar limpia y en buen estado. Se prohíben prendas deslavadas, rotas o con estampados políticos/deportivos.
  • Eventos formales (desayunos, webinars, hackatones):
    • ✅ Pantalón de vestir
    • ✅ Camisa
  • Días sábados o informales:
    • ✅ Mezclilla
    • ✅ Tenis limpios
    • ✅ Playeras tipo polo
    • ❌ Calzado abierto (sandalias, chanclas)
  • Uniforme: Deberá portarse según el calendario de Recursos Humanos una vez entregado.

Accesorios y Cuidado Personal

  • Cinturones con hebillas excesivamente grandes

💻 Estándares de Desarrollo de Software

📌 Nota Importante: Los nuevos lineamientos aplicarán para nuevas funcionalidades y refactorizaciones, no para el código legado.

Prácticas de Código Limpio (Clean Code)

🎯 KISS - Keep It Simple, Stupid

Mantener las soluciones simples y directas. Evitar la complejidad innecesaria.

// ❌ MAL: Complejo e innecesario function verificarEdad(edad) { if (edad >= 18) { return true; } else { return false; } } // ✅ BIEN: Simple y directo function verificarEdad(edad) { return edad >= 18; }
♻️ DRY - Don't Repeat Yourself

No repetir código. Crear servicios reutilizables en lugar de copiar y pegar.

// ❌ MAL: Código duplicado function calcularInteresMensual(monto) { return monto * 0.05; } function calcularInteresAnual(monto) { return monto * 0.05 * 12; } // ✅ BIEN: Reutilizable function calcularInteres(monto, tasa, meses = 1) { return monto * tasa * meses; }
🚫 YAGNI - You Aren't Gonna Need It

No programar para escenarios futuros no solicitados. Cualquier mejora detectada debe documentarse en un Plan de Desarrollo de Software (PDS) para su posterior evaluación.

⚠️ Importante: Si detectas una posible mejora futura, NO la implementes ahora. En su lugar:
  1. Documéntala en el PDS
  2. Justifica su necesidad
  3. Espera aprobación del equipo

Principios SOLID

Se implementarán los principios SOLID en los nuevos desarrollos:

S - Responsabilidad Única (Single Responsibility)

Una clase, una tarea. Cada clase debe tener una sola razón para cambiar.

// ❌ MAL: Hace demasiado class Usuario { guardarEnBD() { } enviarEmail() { } validarDatos() { } } // ✅ BIEN: Responsabilidad única class Usuario { } class UsuarioRepository { guardarEnBD() { } } class EmailService { enviarEmail() { } } class UsuarioValidator { validarDatos() { } }
O - Abierto/Cerrado (Open/Closed)

Abierto a extensión, cerrado a modificación. Puedes agregar funcionalidad sin modificar código existente.

L - Sustitución de Liskov (Liskov Substitution)

Las clases hijas deben poder reemplazar a sus clases padres sin romper la funcionalidad.

I - Segregación de Interfaces (Interface Segregation)

Interfaces pequeñas y específicas. No forzar a las clases a implementar métodos que no usan.

D - Inversión de Dependencias (Dependency Inversion)

Depender de abstracciones/interfaces, no de implementaciones concretas.

Nomenclatura y Autodocumentación

✅ El código debe ser autodocumentado:
  • Los nombres deben explicar el "qué" y el "por qué"
  • Usar verbos y sustantivos claros
  • Evitar nombres genéricos o vagos
// ❌ MAL: No autodocumentado function calc(m, t, p) { return m * t * p; } // ✅ BIEN: Autodocumentado function calcularInteresMoratorio(montoVencido, tasaMora, plazoVencido) { return montoVencido * tasaMora * plazoVencido; }
📝 Regla de Comentarios:

Los comentarios solo deben explicar el "por qué" se tomó una decisión técnica, no el "qué" hace el código.

// ❌ MAL: Explica el QUÉ (obvio) // Suma dos números function sumar(a, b) { return a + b; } // ✅ BIEN: Explica el POR QUÉ // Usamos decimales para evitar errores de redondeo en intereses function calcularInteres(monto, tasa) { return parseFloat((monto * tasa).toFixed(2)); }

Responsabilidades de la Arquitectura

Se definieron estrictamente las capas del backend:

🎮 Controladores (Controllers)
  • Orquestan las solicitudes HTTP
  • Validan la información de entrada
  • Retornan respuestas HTTP apropiadas
  • NO contienen lógica de negocio
⚙️ Servicios (Services)
  • Manejan toda la lógica de negocio
  • Coordinan operaciones entre repositorios
  • Aplican reglas de negocio
  • Transforman datos para presentación
💾 Repositorios (Repositories)
  • Gestionan el acceso a datos
  • Ejecutan queries a la base de datos
  • Mapean entidades a objetos
  • NO contienen lógica de negocio
📦 DTOs (Data Transfer Objects)
  • Transportan datos entre capas
  • Última capa de validación
  • Definen contratos de entrada/salida
  • Protegen la estructura interna
🏛️ Entidades (Entities)
  • Representación exclusiva de tablas de BD
  • Mapeo 1:1 con la base de datos
  • Contienen solo propiedades, sin lógica

Asincronismo y Revisiones de Código

⚡ Regla Obligatoria:

Todos los métodos de entrada y salida de datos (I/O) deben ser asíncronos utilizando async/await.

// ❌ MAL: Síncrono function obtenerUsuario(id) { return db.query('SELECT * FROM users WHERE id = ?', [id]); } // ✅ BIEN: Asíncrono async function obtenerUsuario(id) { return await db.query('SELECT * FROM users WHERE id = ?', [id]); }
✅ Code Reviews Semanales:

Se implementarán sesiones obligatorias de revisión de código (Code Reviews) al final de cada semana, comenzando con los módulos desarrollados recientemente.

  • Cuándo: Viernes antes del cierre
  • Qué revisar: Módulos de la semana
  • Objetivo: Verificar cumplimiento de estándares
